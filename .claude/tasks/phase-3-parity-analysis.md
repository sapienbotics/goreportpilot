# Phase 3 — Design System Parity Analysis

**Status:** analysis only — NO code changes yet.
**Prepared:** 2026-04-20
**Scope:** ground-truth investigation of the 20 concerns raised after Option F v1 deploy. Root causes identified by reading actual code + thumbnails + theme_layout. Architectural decisions flagged for your approval before any fix is coded.

---

## 0. Executive summary

The 20 concerns fall into **5 real defects** plus a large cluster of symptoms that all trace back to a single architectural oversight: the thumbnail PNGs were exported from raw `.pptx` files without substituting placeholder tokens and without removing the template's text/logo shapes. Every overlay is being drawn on top of a PNG that already contains conflicting native content.

**The 5 real defects:**

1. **Thumbnail composition** — PNGs show literal `{{client_name}}`, `{{report_period}}`, and `{{agency_logo}}` tokens (concerns C1, C2, C3, C20).
2. **CSS brand-tint not rendering in preview** — for modern_clean which explicitly supports band tinting, user's hex does not apply to the band overlay (concern C4).
3. **CSS accent-bar not rendering in preview** (concern C5).
4. **Logo overlay ignores user selection** — overlay is using the thumbnail's baked-in logo positions, not the user's dropdown choices (concern C6). Note: backend PPTX logo placement is actually working per Railway logs; the preview-side overlay is where the gap is.
5. **PPTX color lightening** — user hex `#3ab2cb` renders as `~#4cc5d5` in the downloaded PPTX (concern C8). Most likely cause: theme colour inheritance via `<a:lumMod>` / `<a:lumOff>` elements on the template's original `<a:solidFill>` that python-pptx's `fore_color.rgb` setter doesn't clear.

Everything else on the list is either a downstream symptom of one of these 5, or a design-intent discussion (template chrome: "PERFORMANCE REPORT" label, report_type text, footer divider — these appear in PPTX because the template designer put them there; Option F v1 explicitly chose to leave them).

**Four architectural decisions need your approval before coding:**

- **D1.** Thumbnail strategy (Option A vs B)
- **D2.** Template chrome strategy (hide vs show vs delete)
- **D3.** Preview-PPTX fidelity promise
- **D4.** Color transformation fix approach

Details in §4.

## 1. Ground-truthed concerns with code locations

### PREVIEW BUGS (Design tab CSS overlay)

#### C1. Thumbnails show literal `{{client_name}}` / `{{report_period}}` / `{{agency_logo}}` tokens

**Root cause:** Thumbnails in `backend/static/cover_thumbnails/` were generated by LibreOffice CLI (`soffice --headless --convert-to png`) directly from the raw `.pptx` files. Each template's cover slide contains text shapes with literal `{{token}}` strings that haven't been substituted. LibreOffice renders them as-is.

**Evidence:** Running python-pptx on `modern_clean.pptx` produces this shape list for slide 0:
```
[0] Shape 0      — the header band (no text)
[1] Text 1       — "PERFORMANCE REPORT"         ← visible in thumbnail
[2] Text 2       — "Prepared by {{agency_name}}" ← visible in thumbnail
[3] Text 3       — "{{agency_logo}}"             ← visible in thumbnail
[4] Text 4       — "{{client_name}}"             ← visible in thumbnail
[5] Text 5       — "{{report_period}}"           ← visible in thumbnail
[6] Text 6       — "{{report_type}}"             ← visible in thumbnail
[7] Text 7       — "{{client_logo}}"             ← visible in thumbnail
[8] Shape 8      — footer divider (no text)
[9] Text 9       — "{{agency_name}}"             ← visible in thumbnail
```

**Files involved:** `backend/static/cover_thumbnails/*.png` (all 6)
**Severity:** BLOCKER. This is the single cause of C1, C2, C3, C6 (partially), C20.
**Fix:** requires architectural decision D1 (see §4).

#### C2 & C3. User headline / subtitle overlay collides with thumbnail's native placeholder text

**Root cause:** Same as C1. DesignTab.tsx renders `<div>` overlays for headline + period line **on top of** the thumbnail, which already has `{{client_name}}` / `{{report_period}}` text baked in. CSS overlay doesn't mask the thumbnail; it stacks above it. Both appear.

**Code:** `frontend/src/components/clients/tabs/DesignTab.tsx:488-509`

**Fix:** tied to D1 — if thumbnails have placeholder shapes removed, the overlay becomes the sole text source.

#### C4. Preview band tint not rendering for modern_clean despite `brand_tint_strategy: 'header_band'`

**Root cause:** The CSS band style uses `mixBlendMode: 'multiply'` at `opacity: 0.85`. In the screenshot the band appears unchanged (template indigo, not user teal). Two possibilities:
1. `mix-blend-mode: multiply` + the template's dark indigo = an even darker shade that visually doesn't "look like" the user's teal even though the overlay IS present. Visual perception issue.
2. The overlay is positioned correctly but being z-occluded by the thumbnail's inline shapes. Unlikely — `<div style={{position: 'absolute'}}>` after the `<img>` stacks above unless explicit `zIndex` says otherwise.

**Code:** `DesignTab.tsx:458-466` — band style config. `:494-495` — band render.

**Verdict:** CSS logic is sound. The issue is multiply-blend vs dark template fill producing a muddy output that doesn't match user's intent. Switching to a SOLID overlay (no blend) renders the user's colour exactly but hides decorative detail beneath. Trade-off.

**Fix:** switch to solid overlay (no `mixBlendMode`, opacity 1.0) OR use `screen` blend on dark templates / `multiply` on light. Simpler: solid always.

#### C5. Accent bar not visible in preview

**Root cause:** The accent bar style only renders when `layout.header_band` exists AND `brand_tint_strategy === 'header_band'`. For modern_clean both are true, so it SHOULD render. Screenshot shows it missing.

**Code:** `DesignTab.tsx:468-479` (accent style), `:497-498` (render).

Possible cause: the user tested with default accent `#3acbc1` as a hex value. `isValidHex("#3acbc1")` returns true. The style should compute. Need to DevTools-inspect to confirm whether the `<div>` is in the DOM but z-occluded, or the accent state is empty.

**Best theory:** the accent is rendered but sits at `top: 2.12"` (header_band.y + header_band.h - 0.08 = 0 + 2.2 - 0.08 = 2.12) — which is the bottom edge of the band. The thumbnail already has content in that row (template's own divider line), so the thin teal bar may be visually merging with it or being cropped by the `overflow-hidden` container.

**Fix:** verify via DevTools; likely needs higher z-index or different opacity to be distinguishable.

#### C6. Logo position/size dropdowns don't affect preview

**Root cause:** DesignTab DOES pass position + size to the preview (`DesignTab.tsx:513-530`). And `PreviewLogo` DOES compute coordinates from position (`positionToXY`). So the overlay logo should move.

BUT the thumbnail PNG underneath still shows the template's native logos at their hardcoded positions. Result: user sees BOTH — the thumbnail's baked-in placeholder AND the overlay real logo. The "placeholder" text `{{agency_logo}}` is what's visible; the real logo sits where the user picked. They look like "logos haven't moved" because the template text is what catches the eye.

**Code:** `DesignTab.tsx:512-530` (overlay render — works), thumbnail PNG (carries baked placeholder — the real problem).

**Fix:** same as D1 — thumbnails need placeholder shapes removed. Once thumbnails are clean, the overlay-driven logo positions become visible as the ONLY logos.

#### C7. Preview aspect ratio looks compressed

**Root cause:** `DesignTab.tsx:484` uses `style={{ aspectRatio: '13.333 / 7.5' }}`. Modern browsers (Chrome, Edge, Safari 15+) honour `aspect-ratio` CSS. The `<img>` is `object-cover` which may crop.

Actual thumbnails are 1600×900 PNG (16:9 = 1.778:1). Slide spec is 13.333:7.5 (1.778:1). Exactly matches. So the PREVIEW container aspect ratio matches the PNG aspect ratio — should not compress.

**Most likely explanation:** the preview panel sits in a `minmax(0, 480px)` column. At browser widths >= lg breakpoint, the preview is exactly 480×270 px. That's small — text overlays at clamped font sizes may appear cramped. Not a bug per se; a sizing choice.

**Fix:** give preview more width (e.g. 600px) OR scale with a transform if needed. Minor.

### PPTX RENDERING BUGS

#### C8. Brand color `#3ab2cb` renders as lighter `~#4cc5d5` in PPTX

**Root cause (highest probability):** the template's original header-band shape uses `<a:solidFill><a:srgbClr val="XXXXXX"><a:lumMod val="..."/><a:lumOff val="..."/></a:srgbClr></a:solidFill>` XML. When python-pptx's `fill.solid()` + `fore_color.rgb = RGBColor(...)` writes a new `srgbClr val="3AB2CB"`, it may NOT clear the `<a:lumMod>` and `<a:lumOff>` children. Those luminance modifiers then re-apply to the user's hex → lighter output.

**Code:** `backend/services/cover_customization.py:119-128`:
```python
band.fill.solid()
band.fill.fore_color.rgb = RGBColor(*rgb)
```
No explicit clear of child elements.

**Verification needed:** open the rendered PPTX as a ZIP, inspect `ppt/slides/slide1.xml`, find the header-band `<p:sp>` element, look inside `<p:spPr>/<a:solidFill>/<a:srgbClr>`. If `val="3AB2CB"` BUT `<a:lumMod val="75000"/>` or `<a:lumOff val="25000"/>` is still present, theory confirmed.

**Fix:** after setting fill color, explicitly remove any `<a:lumMod>` / `<a:lumOff>` / `<a:shade>` / `<a:tint>` child elements from the new `<a:srgbClr>`. One-time lxml patch in `_recolor_header_band`.

**Alternative cause (lower probability):** WPS Office rendering difference. Unlikely to produce consistent ~10% lightening.

#### C9. Accent `#3acbc1` renders as "bright green"

**Root cause (most likely):** the hex `#3acbc1` itself — decomposing (0x3A, 0xCB, 0xC1) = (58, 203, 193) — is in the cyan-green family. It's "bright green" when rendered on a dark background because the green channel dominates. Not a bug; the user's colour choice is greenish-teal and renders as such.

**Verification needed:** if you type `#008080` (pure teal) into the Accent field, does it render as teal in the PPTX? If yes, C9 is a colour-perception matter, not a bug. If it still renders green, there's a variable-wiring bug.

**Code:** `cover_customization.py:187-216` (_draw_accent_bar). Uses `_hex_to_rgb(hex_str)` on the passed-through accent. No wiring issue I can see statically.

**Fix:** Likely no code fix needed. Document behaviour. If the user confirms that a pure teal hex ALSO renders green, there's something else — would need debug logs.

#### C10. Headline position at y ≈ 3" visually unbalanced

**Root cause:** Template designer placed `{{client_name}}` at y=2.70" in `modern_clean.pptx`. Backend uses the template's coordinate — no reposition (per Option F v1 contract). This IS the template's intended design.

**Code:** `backend/services/theme_layout.py:33` — `client_name_box: {"y": 2.70, ...}`. Measured directly from the template.

**Verdict:** NOT a bug. Design intent. The template designer chose y=2.70" because the header band (y=0 to y=2.2) is dense with text, and the title sits in the mid-body area for visual hierarchy. Different themes place it differently (bold_geometric at y=2.0, gradient_modern at y=3.1).

If this looks "wrong" to the user, the fix is to pick a different theme OR revise the template itself.

#### C11. `{{report_type}}` → "Performance Report" surfaces in PPTX but not preview

**Root cause:** This is template chrome. The template has a text shape with `{{report_type}}` token on the cover. When the generator substitutes placeholders, it fills the token with the actual report type from the config. Preview doesn't show this because thumbnails have the raw token (C1).

**Code:** `report_generator.py` — builds replacements dict including `{{report_type}}`. Substitution is intentional, correct.

**Verdict:** per Option F v1 (DESIGN-SYSTEM-PLAN.md §11 Option A), template chrome is kept. This concern is about the preview NOT showing chrome that the PPTX does show. Gap between the two surfaces.

**Fix:** tied to D2 (template chrome strategy).

#### C12. Agency logo rendered at "top-right" despite user selecting different position

**Root cause:** Without knowing what the user actually selected, and given that Railway logs in Phase A showed `_logo_corner_xy position='top-center' logo_w=... → left=5706058 top=274320` (= 6.24", 0.30" — correct top-center), the backend logo placement is verified-working.

The "rendered at top-right" observation may be:
- Misreading of screenshot (the logo IS at top-center which can look right-leaning if the slide has left-heavy content)
- OR the user selected something other than what they remember
- OR (less likely) there's a flow-through issue for this particular request

**Code:** `report_generator.py:_embed_logos` + `_logo_corner_xy`. Both traced clean in Phase 2 fix v4.

**Verification needed:** re-generate preview, grep Railway logs for `_embed_logos: agency_pos=...`. If logs show `top-center` → backend worked. If logs show `default` or `top-right` → value lost in flight, dig deeper.

**Fix:** log verification first. If logs show correct position, the concern is visual perception. If logs show wrong position, isolate the flow-through break.

#### C13. Client logo oversized despite "default" size selection

**Root cause:** `_logo_max_box(size="default", kind="client")` returns `(3.0", 2.0")` per `report_generator.py:978` — that's the "medium / default" client logo max box. A client logo 2" tall is LARGE on a 7.5" slide.

**Verdict:** the "default" label is misleading — it's actually medium-sized (3"×2"). "Small" is 1.5"×1.0", which is probably what the user actually wanted.

**Code:** `report_generator.py:969-982` — `_logo_max_box()`. The default size for client logos is hardcoded at `(Inches(3.0), Inches(2.0))`.

**Fix:** Either (a) relabel "default" dropdown option to "Medium" and add a smaller "Default" option with e.g. `(2.0", 1.2")`, OR (b) document the current sizes. Minor UX tweak.

#### C14. Footer divider visible in PPTX, not in preview

**Root cause:** Template chrome. `modern_clean.pptx` has Shape 8 at (0.8", 7.0", w=11.7", h=0.015") — a horizontal line divider. It renders in PPTX. Preview thumbnail shows it too BUT the resolution is 1600×900 at a 480×270 display size, so a 0.015"-tall line = 1 pixel at display scale, effectively invisible.

**Code:** no code — template designer's choice.

**Fix:** tied to D2.

## 4. Architectural decisions required

Four decisions. Each has trade-offs. **Pick one path per decision** and I'll implement accordingly.

### D1. Thumbnail strategy

**Option A — "Sample-substituted thumbnails"**
- Build a script that loads each `.pptx`, substitutes `{{client_name}}` → "Client Name", `{{report_period}}` → "April 2026", `{{agency_name}}` → "Your Agency", etc., saves as a temporary PPTX, exports to PNG via LibreOffice.
- User still sees overlays on top, but thumbnail's text is readable placeholder strings ("Client Name") instead of curly-brace tokens.
- **Pro:** thumbnails look polished on the picker cards (no ugly `{{...}}`).
- **Con:** overlay still stacks on top of substituted text, creating double-text when user provides a custom headline. Headline "Test Header" overlays "Client Name" → unreadable blur.

**Option B — "Chrome-only thumbnails" ⭐ RECOMMENDED**
- Build a script that loads each `.pptx`, DELETES every text-bearing shape AND every logo placeholder shape from the cover, saves as a temp PPTX, exports to PNG.
- Thumbnails show ONLY the design chrome: header band fill, decorative shapes, background colour, dividers.
- All text and logos come from CSS overlays. One visual layer of truth per element.
- **Pro:** no text or logo overlap possible. WYSIWYG preview.
- **Con:** some themes rely on text density for balance (minimal_elegant is mostly whitespace — a chrome-only thumbnail would be nearly blank). Overlay must carry full visual weight.

**Option C — "Ditch thumbnails, build cover in pure CSS"**
- Render cover as a pure `<div>` composition using theme_layout coordinates + colour constants. No PNG base.
- **Pro:** no LibreOffice dependency for preview generation. Instant updates.
- **Con:** loses the template designer's decorative artwork (gradients, geometric shapes, lines). CSS can't replicate arbitrary vector decoration. Would require re-authoring each theme's decoration in CSS — significant work.

**My recommendation: Option B.** It aligns with Option F v1's principle ("template owns design, code owns overrides") while giving the preview a clean canvas. The chrome-only thumbnail still carries the template's colour palette, header band shape, and decorative design — everything that makes `modern_clean` look different from `dark_executive`. Text and logos are owned entirely by the overlay layer. For `minimal_elegant`, the "empty" thumbnail IS correct — the whitespace IS the design.

### D2. Template chrome strategy (PERFORMANCE REPORT label, report_type, divider line, Prepared-by line)

**Option A — Accept and document**
- Keep template chrome as-is. Surface it in the preview too (via Option B thumbnails + secondary overlay for template-chrome text like "PERFORMANCE REPORT").
- Preview matches PPTX. User sees everything before downloading.
- **Pro:** honest. Zero template edits. Preview = reality.
- **Con:** the preview is busier. Design tab shows "PERFORMANCE REPORT" etc. on every theme card.

**Option B — Strip chrome from templates ⭐ RECOMMENDED**
- One-time manual PowerPoint edit per template (6 files, ~30 min total):
  - Delete "PERFORMANCE REPORT" text shape
  - Delete "Prepared by {{agency_name}}" text shape
  - Delete `{{report_type}}` text shape
  - Delete the thin footer divider line
  - Leave: header band, `{{client_name}}`, `{{report_period}}`, `{{agency_name}}` footer, `{{agency_logo}}`/`{{client_logo}}` placeholders (for sizing reference)
- Regenerate chrome-only thumbnails (D1 Option B).
- PPTX becomes cleaner — only user-relevant content.
- **Pro:** final PPTX is minimal + professional. Matches the "clean" design spirit the preview has been trying to fake.
- **Con:** template edits are manual work. If a designer later updates templates, these strips need to be re-applied.

**Option C — Hide chrome via code**
- At generate time, find and hide/delete chrome shapes programmatically.
- **Pro:** no manual template edits.
- **Con:** reintroduces the shape-stripping pattern that caused 10 iterations of bugs in phases 3 v2..v10. Strong do-not-do.

**My recommendation: Option B.** Three hours of one-time PowerPoint work eliminates an entire class of parity bugs forever. Option A is also acceptable if template edits aren't feasible.

### D3. Preview-PPTX fidelity promise

**Option A — "Pixel-identical" promise (current, unachieved)**
- Preview and PPTX must look the same at the pixel level.
- **Con:** impossible — Chrome font rendering ≠ PowerPoint/WPS font rendering; aspect-ratio handling differs; sub-pixel anti-aliasing differs.
- Current symptom: users file bugs every time preview ≠ PPTX by even a pixel.

**Option B — "Meaningful representation" promise ⭐ RECOMMENDED**
- Preview shows the EXACT same content as the PPTX: same theme, same headline, same subtitle, same colours, same logo positions, same presence/absence of each element.
- Pixel-exact rendering is NOT promised. Font metrics, line spacing, anti-aliasing may differ.
- "Download PPTX preview" button exists for byte-accurate verification.
- Documentation in the Design tab makes this explicit: *"Preview approximates the rendered cover. Click Download PPTX preview for the exact file."*
- **Pro:** achievable. Aligns with how every adjacent SaaS (Figma → PDF export, Notion → Word export) treats their preview.
- **Con:** users who want pixel-exactness have to download. 1 extra click.

**My recommendation: Option B.** It's the only honest contract. State it prominently in the Design tab and in any marketing copy about the feature.

### D4. Color transformation fix

**Theory (C8):** User hex `#3AB2CB` becomes `~#4CC5D5` because the template's original `<a:srgbClr>` has `<a:lumMod>` or `<a:lumOff>` or `<a:tint>` or `<a:shade>` child elements that modify the final rendered colour. python-pptx's `fore_color.rgb` setter writes a new `val` attribute but may not strip the modifier children.

**Option A — XML-scrub after setting fill ⭐ RECOMMENDED**
- In `_recolor_header_band`, after setting `band.fill.fore_color.rgb`, explicitly find the `<a:srgbClr>` element and remove any `<a:lumMod>`, `<a:lumOff>`, `<a:tint>`, `<a:shade>`, `<a:alpha>`, `<a:gamma>` child elements.
- ~10 lines of lxml code.
- **Pro:** guarantees exact colour rendering across all renderers (PowerPoint, WPS, Google Slides, LibreOffice).
- **Con:** requires `lxml` (already a dep). Needs verification by inspecting the actual output XML.

**Option B — Rebuild the shape's fill from scratch**
- Instead of mutating the existing `<a:solidFill>`, remove it entirely and add a fresh one.
- **Pro:** cleanest.
- **Con:** more invasive. May remove other fill properties the designer intended.

**Option C — Document and accept**
- Accept that the rendered colour may differ from the user's hex by up to 15% due to template colour inheritance.
- **Con:** user-visible defect that breaks the "what you picked is what you get" promise.

**My recommendation: Option A (XML-scrub).** Targeted, low-risk, directly addresses the root cause. Needs one verification step: produce a test PPTX with a known hex, unzip, inspect `slide1.xml`, confirm the srgbClr element has no modifier children.

Accent color C9 (green rendering) is probably NOT a bug — user's chosen hex `#3acbc1` is a cyan-green that may just read as "green" against dark context. Fix: confirm with user by testing pure teal `#008080` before adding code. If #008080 also renders green, there's a real bug.
